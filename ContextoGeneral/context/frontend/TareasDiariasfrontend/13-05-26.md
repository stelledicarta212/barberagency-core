# Tareas Frontend - 13-05-26

## Objetivo del día
Corregir el flujo `Registro Barbería -> Editor Landing` para que los datos heredados (servicios y barberos) no queden solo en semilla local, sino que se sincronicen en tablas canónicas antes de redirigir al editor.

## Cambios realizados hoy

### 1) Registro Barbería: sincronización canónica por RPC
Archivo: `project/templates/plantillas/registrobarberia.html`

- Se configuró RPC canónica por defecto:
  - `DEFAULT_CANONICAL_SYNC_RPC = "ba_sync_registro_collections"`
  - override opcional por runtime: `window.BA_CANONICAL_SYNC_RPC`.
- Se mantuvo base PostgREST:
  - `POSTGREST_BASE` (default `https://api.agencia2c.cloud`).
- Se eliminó el enfoque principal de inserts directos de frontend a `/servicios` y `/barberos`.
- Se implementó `syncCanonicalCatalogs(data, payload)` para llamar:
  - `POST /rpc/ba_sync_registro_collections`
  - body:
    - `p_barberia_id`: id real devuelto por onboarding.
    - `p_servicios`: servicios normalizados.
    - `p_barberos`: barberos normalizados.

### 2) Normalización de payload heredado
Archivo: `project/templates/plantillas/registrobarberia.html`

- Servicios (`normalizeServiceDraftRows`):
  - salida: `{ nombre, duracion_min, precio }`
  - defaults: `duracion_min=30`, `precio=0`.
  - se excluyen nombres vacíos.
- Barberos (`normalizeBarberDraftRows`):
  - salida: `{ nombre }`
  - se excluyen nombres vacíos.

### 3) Regla de continuidad del flujo
Archivo: `project/templates/plantillas/registrobarberia.html`

- `submitDraft()` y `submitDraftAndContinue()` ahora ejecutan:
  1. `postOnboardingPayload(payload)`
  2. `syncCanonicalCatalogs(data, payload)`
  3. recién después `persistLandingSeed(...)` + redirección.
- Si RPC falla, no se debe considerar éxito funcional.
- Mensaje de error de usuario estandarizado:
  - `No se pudieron guardar servicios y barberos en la base de datos. Revisa permisos o RPC.`

### 4) Manejo de error funcional RPC
Archivo: `project/templates/plantillas/registrobarberia.html`

- Si RPC retorna `{ ok:false, error:"barberia_no_encontrada" }`, se muestra error claro y se aborta continuidad.
- Se conserva fallback visual bajo flag:
  - `BA_ALLOW_LOCAL_SEED_FALLBACK=1`
  - solo emergencia, no flujo principal.

### 5) Editor Landing: contexto de identidad
Archivo: `project/templates/editor/landing_editor_v2_unico_vscode.html`

- Se ajustó lógica de identidad para evitar inferencias débiles desde slug cuando no corresponde.
- Objetivo: disminuir casos de publish con id inconsistente (`barberia_no_encontrada`).
