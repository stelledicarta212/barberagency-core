# Tarea del dia - 31-05-26

## Tema principal

Sesion real de usuario y apertura automatica de su barberia en produccion.

Esta tarea queda como prioridad principal. No se debe avanzar a otra tarea hasta cerrar este flujo con pruebas reales.

---

## Objetivo

Cuando un usuario se registre con Google, cree una barberia y luego vuelva a entrar al sistema, BarberAgency debe reconocerlo por sesion segura y llevarlo automaticamente a su barberia real, con sus datos reales.

Flujo esperado:

```txt
Usuario inicia sesion con Google
  -> backend valida Google
  -> backend crea/actualiza usuario
  -> backend crea cookie segura ba_session
  -> usuario crea barberia
  -> barberia queda asociada a ese usuario
  -> al volver a entrar, session/me resuelve su barberia
  -> dashboard/inicio carga esa barberia
```

---

## Hallazgos actuales

### 1) Login Google ya tiene una base util

Workflow activo en produccion:

```txt
BarberAgency - Auth Google Session v1
Path: barberagency/auth/google-session
Estado: activo
```

Hallazgos:

- Recibe `id_token` desde WP.
- Valida el token contra Google.
- Crea o actualiza usuario en `public.usuarios`.
- Crea cookie segura:

```txt
ba_session=...; HttpOnly; Secure; SameSite=Lax
```

Conclusion:

```txt
El login Google sirve y no se debe botar.
```

---

### 2) Falta cerrar `session/me`

Workflows encontrados:

```txt
BarberAgency - Session Me v1
BarberAgency - Session Me v2
```

Estado encontrado:

```txt
inactivos
```

Prueba realizada:

```txt
GET https://barberagency-n8n.gymh5g.easypanel.host/webhook/barberagency/session/me
```

Resultado:

```txt
404 Not Found
```

Conclusion:

```txt
WP ya intenta usar session/me, pero en produccion ese endpoint no esta activo.
```

---

### 3) `session/me` actual no devuelve barberia

El workflow `Session Me v2` encontrado solo devuelve datos del usuario:

```txt
user_id
email
nombre
plan_id
puede_crear_barberia
```

Pero falta devolver:

```txt
current_barberia
barberia_id
slug
nombre de barberia
role
barberias disponibles
```

Conclusion:

```txt
Aunque se active, todavia no resuelve la barberia del usuario.
```

---

### 4) Registro de barberia ya sincroniza fuente de verdad, pero falta amarrarlo a la sesion

Workflow activo:

```txt
registro_barberia
Path: registro-barberia
Estado: activo
```

Hallazgos:

- Crea barberia en `public.barberias`.
- Sincroniza servicios, horarios y barberos.
- Crea/actualiza admin en `public.usuarios`.
- Asigna:

```sql
barberias.owner_id = usuarios.id
```

Riesgo:

```txt
Actualmente depende del correo del admin enviado en el formulario.
Si el correo del admin no coincide con el usuario logueado por Google, la barberia puede quedar asociada a otro usuario.
```

Conclusion:

```txt
El registro debe usar la sesion autenticada como autoridad principal.
```

---

## Fuente de verdad definida

Para produccion, la fuente de verdad debe ser PostgreSQL:

```txt
public.usuarios
public.barberias.owner_id
public.barberos.usuario_id
public.servicios
public.citas
public.clientes
public.subscriptions
```

El navegador no debe ser fuente de verdad.

Permitido como cache visual:

```txt
localStorage ba_user_nombre
localStorage ba_user_email
localStorage ba_plan_name
```

No permitido como autoridad:

```txt
localStorage ba_user_id
localStorage ba_puede_crear_barberia
localStorage ba_plan_status
localStorage ba_google_token
slug de URL sin validacion backend
```

---

## Solucion propuesta para produccion

### Decision de arquitectura: `/inicio/` como portal privado

La pagina `/inicio/` debe ser el centro privado del usuario despues del login.

No se debe depender de la pagina `Nosotros` para que el usuario encuentre el boton de crear barberia. `Nosotros` puede seguir siendo una pagina publica/marketing, pero no debe ser parte obligatoria del flujo operativo.

Flujo recomendado despues de la tarjeta de bienvenida:

```txt
Usuario se registra con Google
  -> tarjeta de bienvenida
  -> redireccion a /inicio/
  -> /inicio/ consulta session/me
  -> /inicio/ decide el destino correcto
```

Reglas de destino:

```txt
Usuario sin sesion:
  mostrar login/registro

Usuario con sesion y 0 barberias:
  mostrar "Crea tu primera barberia"
  boton: Crear barberia
  destino: /registro-barberia/

Usuario con sesion y 1 barberia:
  llevar al dashboard de esa barberia

Usuario con sesion y 2 o mas barberias:
  mostrar pantalla "Elige tu barberia"
  tarjetas/lista de barberias
  boton en cada una: Entrar
```

Decision:

```txt
Para varias barberias, primero se usara selector obligatorio "Elige tu barberia".
Mas adelante se puede mejorar con "recordar ultima barberia usada", pero no debe ser el primer paso.
```

El boton publico del sitio debe cambiar segun estado:

```txt
Sin sesion:
  Registrarme / Crear mi barberia

Con sesion y sin barberia:
  Crear barberia

Con sesion y con una barberia:
  Ir a mi panel

Con sesion y varias barberias:
  Ir a mis barberias
```

### Paso 1) Activar/corregir `session/me`

Endpoint esperado:

```txt
GET /webhook/barberagency/session/me
```

Debe:

1. Leer cookie `ba_session`.
2. Validar el token de sesion.
3. Resolver `user_id`.
4. Consultar `public.usuarios`.
5. Consultar la barberia del usuario en `public.barberias`.
6. Responder con usuario y barberia activa.

Respuesta esperada:

```json
{
  "ok": true,
  "user": {
    "id": 10,
    "email": "usuario@gmail.com",
    "nombre": "Carlos"
  },
  "current_barberia": {
    "id": 185,
    "slug": "barberia-185",
    "nombre": "Barberia 74",
    "role": "owner"
  },
  "barberias": [
    {
      "id": 185,
      "slug": "barberia-185",
      "nombre": "Barberia 74",
      "role": "owner"
    }
  ]
}
```

---

### Paso 2) Resolver barberia por usuario autenticado

Regla base:

```sql
SELECT id, slug, nombre
FROM public.barberias
WHERE owner_id = :user_id
  AND deleted_at IS NULL
ORDER BY id DESC;
```

Si hay una barberia:

```txt
abrir automaticamente esa barberia
```

Si hay varias:

```txt
usar ultima barberia o mostrar selector
```

Si no hay ninguna:

```txt
llevar a crear barberia
```

---

### Paso 3) Corregir `registro_barberia`

El registro debe asociar la barberia al usuario de la sesion:

```txt
cookie ba_session -> user_id -> barberias.owner_id
```

El correo escrito en el formulario puede servir como dato de contacto o admin, pero no debe mandar sobre la sesion autenticada.

Regla de produccion:

```txt
El dueno real de la barberia es el usuario autenticado.
```

---

### Paso 4) Ajustar WP `/inicio/`

Al cargar `/inicio/`, WP debe llamar:

```js
fetch("https://barberagency-n8n.gymh5g.easypanel.host/webhook/barberagency/session/me", {
  credentials: "include",
  cache: "no-store"
})
```

Con la respuesta:

- Si `ok = true` y hay `current_barberia`, abrir/cargar esa barberia.
- Si `ok = true` y no hay barberia, llevar a crear barberia.
- Si `ok = false`, limpiar cache local y mandar a login/registro.

---

### Paso 5) Ajustar dashboard/panel

El dashboard no debe decidir identidad desde `localStorage`.

Debe usar:

```txt
session/me -> current_barberia.id -> dashboard/state
```

Todo acceso a datos debe quedar validado por backend.

## Pruebas obligatorias antes de cerrar

Todas las pruebas deben hacerse en Postman o equivalente antes de dar la tarea por terminada. No basta con que el flujo se vea bien en pantalla.

La implementacion debe quedar limpia para produccion:

- Sin soluciones temporales.
- Sin depender de `localStorage` como fuente de verdad.
- Sin exponer tokens sensibles al navegador.
- Sin saltarse validaciones de sesion.
- Sin permitir acceso a barberias ajenas.
- Sin perder reglas de seguridad por rapidez.

### Prueba 1) Login

```txt
Iniciar sesion con Google
Verificar Set-Cookie ba_session
Verificar respuesta ok
```

### Prueba 2) Session me

```txt
GET /session/me con cookie real
Debe devolver usuario y current_barberia
```

Validaciones Postman:

```txt
Request con cookie valida -> 200 ok
Request sin cookie -> 401
Request con cookie invalida -> 401
No debe devolver session_token en el body
No debe devolver ba_google_token
```

### Prueba 3) Crear barberia

```txt
Crear una barberia de prueba
Confirmar en BD que barberias.owner_id = user_id autenticado
```

Validaciones Postman:

```txt
Crear barberia con sesion valida -> owner_id correcto
Crear barberia sin sesion -> 401 o error controlado
Crear barberia con sesion de otro usuario -> no debe asociarse al usuario equivocado
```

### Prueba 4) Volver a entrar

```txt
Cerrar/abrir navegador o usar nueva pestana
Entrar a /inicio/
Debe cargar automaticamente la misma barberia
```

### Prueba 5) No mezclar usuarios

```txt
Usuario A no debe ver barberia de Usuario B
Usuario B no debe ver barberia de Usuario A
```

Validaciones Postman:

```txt
Cookie Usuario A consultando barberia Usuario B -> 403 o error controlado
Cookie Usuario B consultando barberia Usuario A -> 403 o error controlado
```

### Prueba 6) Dashboard

```txt
Dashboard debe cargar datos por current_barberia.id:
- servicios
- barberos
- citas
- clientes
- plan
- QR
- landing
```

Validaciones Postman:

```txt
dashboard/state con barberia del usuario -> 200 ok
dashboard/state con barberia ajena -> 403 o error controlado
dashboard/state sin sesion cuando sea privado -> 401
```

## Seguridad obligatoria

Antes de cerrar, verificar:

```txt
Cookie ba_session:
- HttpOnly
- Secure
- SameSite=Lax o Strict
- Max-Age definido
```

```txt
Respuesta de auth/session:
- No exponer token interno de sesion
- No exponer Google id_token
- No confiar en user_id enviado desde frontend
- No confiar en plan enviado desde frontend
- No confiar en barberia_id enviado desde frontend sin validar permisos
```

```txt
Backend:
- Verifica sesion antes de crear barberia
- Verifica sesion antes de cargar dashboard privado
- Verifica owner_id o rol antes de devolver datos
- Devuelve errores controlados 401/403/400
```

## Criterios de cierre

La tarea solo se considera terminada cuando:

- `session/me` este activo en produccion.
- `session/me` devuelva usuario y barberia real.
- La barberia creada quede asociada al usuario autenticado.
- `/inicio/` abra la barberia correcta sin depender de localStorage.
- Dashboard cargue datos de la barberia correcta.
- Se pruebe con una barberia fija de pruebas sin crear una nueva cada vez.
- Se confirme que otro usuario no puede cargar barberias ajenas.
