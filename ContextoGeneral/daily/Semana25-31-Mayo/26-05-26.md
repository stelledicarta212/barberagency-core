## 🛠️ Sistema de Recuperación de Contraseña Auto-servicio (Completado)

Hemos implementado de forma segura y end-to-end el flujo de recuperación de contraseña auto-servicio para los dueños y barberos de BarberAgency:

### 1) Base de Datos (PostgreSQL)
- Creada la tabla `public.password_resets` para almacenar tokens criptográficos temporales:
  ```sql
  CREATE TABLE IF NOT EXISTS public.password_resets (
    id SERIAL PRIMARY KEY,
    usuario_id INT NOT NULL REFERENCES public.usuarios(id) ON DELETE CASCADE,
    token UUID NOT NULL DEFAULT gen_random_uuid(),
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    used_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
  );
  CREATE UNIQUE INDEX IF NOT EXISTS ix_password_resets_token ON public.password_resets(token);
  ```

### 4) Resolución de Error al Guardar Descanso (26 de Mayo, 2026 - Continuidad)
- **Diagnóstico:** Se identificó que al intentar guardar un descanso en la barbería de pruebas (ID 181), el webhook de n8n reportaba un error de clave foránea: `insert or update on table "barberos_descansos" violates foreign key constraint "barberos_descansos_barbero_id_fkey" (Key (barbero_id)=(0) is not present in table "barberos")`. Adicionalmente, tras corregir este comportamiento en el frontend, el navegador reportaba un error de `Failed to fetch` al intentar guardar el descanso.
- **Causa Raíz:** 
  1. En `src/app/barberos/page.tsx`, la función de ayuda `text(value)` estaba definida como `return typeof value === "string" ? value.trim() : "";`. Dado que la API de dashboard state ahora retorna el `id` del barbero como tipo numérico (`394`, `395`...), `text(item.id)` devolvía una cadena vacía `""`. Esto disparaba un fallback a `"barber-${index+1}"` en los IDs de las tarjetas de barberos en el frontend. Al invocar `toggleOffDay`, se hacía `Number(calendarBarberId)` (es decir, `Number("barber-1")`), lo cual producía `NaN`. Dicho `NaN` se serializaba en JSON como `null`, que al insertarse en base de datos causaba que Postgres interpretara el ID como `0` o generara una violación de clave foránea.
  2. En el workflow de n8n `BarberAgency - Dashboard Barberos Admin` (ID: `h3JdyaI26GbRqrzE`), el nodo de Webhook estaba configurado con el método HTTP `*`. Sorprendentemente, n8n interpreta esto de forma literal como un método personalizado llamado `*` en lugar de una regla comodín ("cualquier método"). Por lo tanto, las solicitudes legítimas de tipo `POST` y `OPTIONS` desde el navegador fallaban con `404 Webhook not registered`, lo cual el navegador reportaba de inmediato como un error de red `Failed to fetch` debido a la falla en la negociación CORS.
- **Solución:**
