# 02-06-26 - Evidencia de registro, fuente de verdad y panel

## Objetivo del dia

Cerrar el flujo de registro de barberias para produccion y dejarlo alineado con la fuente de verdad.

La regla principal queda definida asi:

- El usuario real sale de la sesion `ba_session`.
- La barberia pertenece al usuario autenticado mediante `public.barberias.owner_id`.
- `session/me` es la fuente para saber que barberias tiene el usuario.
- `Mis barberias` no debe usar datos inventados ni `localStorage` como fuente canonica; solo cachea lo que devuelve `session/me`.
- Al registrar, editar o publicar, no se debe cambiar `id`, `slug` ni `owner_id` de una barberia ya creada.

## 1) Registro de barberia y fuente de verdad

### Problema encontrado

Al reenviar/publicar una barberia existente, el workflow de registro podia crear otra barberia nueva con otro `id` y otro `slug`.

Caso detectado:

- Barberia original: `id = 196`, `slug = barberia-prueba-3`
- Reenvio anterior creo otra: `id = 197`, `slug = barberia-prueba-3-5b8f92`

Esto era delicado porque podia separar plan, dashboard, reservas, barberos y owner real.

### Solucion aplicada

Se corrigio el workflow n8n `registro_barberia` para que:

- Si llega `barberia_id` y pertenece al usuario de sesion, actualice esa misma barberia.
- Si no llega `barberia_id`, cree una nueva barberia.
- Si llega `barberia_id` pero no pertenece al usuario de sesion, responda error.
- Guarde campos base en `public.barberias`:
  - `owner_id`
  - `slot_min`
  - `direccion`
  - `ciudad`
  - `telefono`
  - `email_contacto`
- Mantenga estable:
  - `id`
  - `slug`
  - `owner_id`

### Archivos/scripts relacionados

- `scratch/patch_registro_barberia_session_owner.js`
- `project/templates/plantillas/registrobarberia.html`

### Cambios en `registrobarberia.html`

Se ajusto el frontend del registro para:

- Enviar la sesion con `credentials: "include"`.
- Evitar doble escritura directa desde el frontend a PostgREST/RPC.
- Dejar que n8n haga la escritura canonica.
- Validar que cada barbero tenga un correo diferente.

Motivo de la validacion:

- Cada barbero debe poder ingresar a su panel.
- Si dos barberos comparten correo, la cuenta queda ambigua.
- Produccion debe mostrar un mensaje claro al usuario antes de guardar.

## 3) Sesion del usuario y menu de barberias

### Problema a resolver

El usuario necesitaba saber como entrar a su barberia o al listado de sus barberias desde la tarjeta `Hola, Carlos`.

### Regla definida

El menu debe comportarse asi:

- Si `barberias_count === 0`: mostrar `Crear barberia`.
- Si `barberias_count === 1`: mostrar `Ir a mi barberia`.
- Si `barberias_count > 1`: mostrar `Mis barberias`.

La fuente de verdad es:

```txt
GET /webhook/barberagency/session/me
```

No se debe inventar la barberia desde el frontend.

### Evidencia de sesion

La respuesta de `session/me` mostro:

- `ok: true`
- `user_id: 7`
- `barberias_count: 8`
- `current_barberia.id: 198`
- `current_barberia.slug: barberia-prueba-4`
- `current_barberia.nombre: Barberia Prueba 4`
- `role: owner`
- `next_action: choose_barberia`

Esto confirma que la barberia quedo amarrada al usuario.

### Archivo creado

Se creo:

```txt
boton.html
```

Contenido:

- HTML + JS de la tarjeta `Hola, Carlos`.
- Lectura de sesion desde `session/me`.
- Render de `Mis barberias`.
- Link al dashboard correcto en `barberagency-app`.
- Barberia actual marcada con clase `is-current`.
- CSS manejado aparte.

Destino correcto para dashboard:

```txt
https://barberagency-app.gymh5g.easypanel.host/barberia?slug=barberia-prueba-4
```

Nota:

- El host de WordPress no debe usarse para dashboard.
- WordPress es web publica.
- `barberagency-app` es dashboard/panel.
