# Sincronizar Registro Barbería con servicios/barberos en BD

Actúa como desarrollador senior full-stack en el proyecto BarberAgency.

Tarea:
Sincronizar Registro Barbería con servicios/barberos en BD.

Contexto:
Actualmente Registro Barbería muestra correctamente los datos en el frontend/editor como semilla o inheritedCollections, pero esos datos no siempre quedan persistidos en las tablas reales de PostgreSQL.

Ejemplo del problema:
En el editor se ve:

- Semilla cargada: ID 117
- Servicios: barba, afeitado, corte
- Barberos: barber61, berber61

Pero en BD:

SELECT COUNT(*)
FROM public.servicios
WHERE barberia_id = 117
AND activo = true;

devuelve 0.

Y también:

SELECT COUNT(*)
FROM public.barberos
WHERE barberia_id = 117
AND activo = true;

devuelve 0.

Eso rompe el flujo de publicación porque ba_publicar_barberia(117) responde errores como faltan_servicios_activos o faltan_barberos_activos.

Objetivo:
Corregir Registro Barbería para que los datos que el usuario agrega en el registro queden guardados en las tablas canónicas:

- public.barberias
- public.servicios
- public.barberos
- public.horarios, si aplica

Estructura correcta:
Primero debe existir la barbería en:

public.barberias

Ejemplo:

id = 117
nombre = Barberia 61
slug = barberia-61

Luego todos los datos relacionados deben usar ese mismo id real:

servicios.barberia_id = 117
barberos.barberia_id = 117
horarios.barberia_id = 117

No se debe usar:
- el número visible del nombre “Barberia 61”
- usuario_id
- tenantId temporal
- id de otra barbería
- valores de preview
- inheritedCollections como única fuente persistente

Archivo principal a revisar:
project/templates/plantillas/registrobarberia.html

Función principal a revisar:
submitDraftAndContinue()

RPC ya creada en BD:

{
